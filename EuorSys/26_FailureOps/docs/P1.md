# P1: Lightweight QEC Backend

## Purpose

P1 connects FailureOps to a lightweight QEC-backed experiment while preserving
the P0 attribution workflow.

P1 uses:

```text
Stim-generated repetition-code memory circuit
        ↓
detector-event sampling
        ↓
PyMatching decoding
        ↓
FailureOps interventions and attribution metrics
```

This stage is still intentionally small. It is not a full surface-code
simulation, threshold experiment, decoder paper, or FTQC runtime simulator.

The purpose is to show that FailureOps can analyze logical failure behavior from
a real detector-event backend, not only from the P0 toy event model.

## Run The Full P1 Pipeline

From the repository root:

```bash
conda run -n failureops --no-capture-output python scripts/05_generate_qec_runs.py \
  --num-shots 1000 \
  --distance 5 \
  --num-rounds 8 \
  --data-error-rate 0.03 \
  --measurement-error-rate 0.02 \
  --idle-error-rate 0.002 \
  --decoder-timeout-base-rate 0.01 \
  --seed 42 \
  --output data/results/p1_baseline_runs.csv

conda run -n failureops --no-capture-output python scripts/06_apply_qec_interventions.py \
  --input data/results/p1_baseline_runs.csv \
  --output data/results/p1_intervened_runs.csv

conda run -n failureops --no-capture-output python scripts/03_analyze_attribution.py \
  --input data/results/p1_intervened_runs.csv \
  --output data/results/p1_attribution_summary.csv

conda run -n failureops --no-capture-output python scripts/04_plot_results.py \
  --input data/results/p1_attribution_summary.csv \
  --output figures/p1_intervention_delta_lfr.png \
  --title "FailureOps P1 intervention sensitivity"
```

Expected row counts for the command above:

```text
p1_baseline_runs.csv:       1000 data rows
p1_intervened_runs.csv:     9000 data rows
p1_attribution_summary.csv:    9 data rows
```

## Baseline Records

`p1_baseline_runs.csv` stores one repetition-code memory shot per row.

Important fields:

```text
backend
code_family
distance
num_rounds
shot_id
seed
data_error_rate
measurement_error_rate
idle_error_rate
detector_count
detector_event_count
detector_events
observable_flip
decoder_prediction
qec_decoder_failure
decoder_timeout
decoder_delay
idle_exposure
runtime_idle_flip
logical_failure
failure_mode
failure_pattern
```

`detector_events` is a JSON list of detector indices that fired in that shot.

`observable_flip` is the sampled logical observable from Stim.

`decoder_prediction` is the observable correction predicted by PyMatching.

`qec_decoder_failure` records whether PyMatching's prediction differs from the
sampled observable before adding the toy runtime effects.

`logical_failure` is the final FailureOps outcome after applying the simple
runtime model.

## Runtime Model

P1 adds a minimal runtime layer on top of the decoded QEC shot.

The runtime layer computes:

```text
decoder_timeout
decoder_delay
idle_exposure
runtime_idle_flip
```

These are simple deterministic post-processing fields derived from:

```text
detector_event_count
distance
num_rounds
idle_error_rate
shot seed
```

This is not a full decoder scheduler. It only gives P1 enough system-side
structure to compare QEC noise factors with runtime factors.

## Interventions

P1 includes noise-side interventions:

```text
remove_data_noise
weaken_data_noise_50pct
remove_measurement_noise
weaken_measurement_noise_50pct
remove_idle_noise
weaken_idle_noise_50pct
```

P1 also includes runtime-side interventions:

```text
remove_decoder_timeout
reduce_decoder_delay_50pct
reduce_idle_exposure_50pct
```

## Pairing Semantics

Runtime-side interventions are strictly paired:

```text
same shot_id
same seed
same detector event record
same observable
same decoder prediction
```

Only the runtime factor is changed and the final failure behavior is recomputed.

Noise-side interventions are seed-stable backend comparisons:

```text
same shot_id
same batch seed
same circuit configuration except the intervened noise parameter
```

Stim does not expose a simple event-replay API for deleting one sampled physical
event category from an already sampled detector record. Therefore P1 does not
claim exact physical-event replay for noise-side interventions.

This is still useful for P1, but it is weaker than the ideal paired
counterfactual rule. Later versions should improve this with explicit event
replay or a backend that exposes sampled error mechanisms.

## Current Example Result

With the default command above, the current generated attribution table has:

```text
baseline_logical_failure_rate = 0.033
```

The strongest current intervention is:

```text
remove_decoder_timeout:
  absolute_delta_lfr = -0.030
```

Other interventions with visible effects include:

```text
remove_data_noise
remove_measurement_noise
reduce_decoder_delay_50pct
reduce_idle_exposure_50pct
```

The exact numeric values are stochastic and backend-parameter dependent. The
important P1 result is that a real detector-event backend can feed the same
FailureOps attribution table used by P0.

## Current Limitations

P1 does not yet implement:

```text
surface-code memory experiments
operation-level logical workloads
exact physical-event replay for noise-side interventions
real decoder timeout measurements
real runtime queueing or scheduling
failure pattern shift plots
multi-distance sweeps
threshold-style analysis
```

The runtime model is intentionally minimal and should not be interpreted as a
real FTQC runtime.

## Next Steps

Natural next steps:

```text
add a small surface-code memory backend
record richer detector coordinates
separate backend logical failure from runtime-induced failure
add failure-pattern shift figures
add case studies for decoder timeout and idle exposure stress
make noise-side interventions replay exact sampled error mechanisms
```

The central constraint should remain the same:

> Add only what helps measure which intervenable factors logical failure behavior is sensitive to.
