# P10: EuroSys Main Evidence Pass

## Purpose

P10 addresses the three main evidence gaps for a EuroSys-style submission:

```text
baseline comparison
real-data robustness
measured-runtime intervention closure
paired statistical significance
claim-to-evidence audit
external public-dataset replication
```

P10 is not a new FailureOps mechanism. It is an evaluation pass over P7.5 and
P8 artifacts.

## Baseline Comparison

P10 compares FailureOps paired sensitivity against:

```text
plain baseline LFR
static detector burden
plain aggregate LFR delta
unpaired bootstrap estimator
```

The goal is not to invent weaker baselines. The goal is to show which parts of
the evidence are only available when paired interventions are preserved:

```text
rescued and induced transitions
paired estimator variance
intervention-sensitive rankings
```

## Real-Data Robustness

P10 reports:

```text
sign consistency across real-data conditions
cycle-level sensitivity trend
cycle-rank stability across basis and control-mode groups
paired-vs-unpaired variance robustness
decoder-prior CI checks
optional Google RL QEC v2 same-family expansion checks
optional Google decoder-priors same-family prior sweep
```

These checks support robustness over the observed real-data condition matrix.
They do not claim external replication on independent QEC datasets.

## Measured-Runtime Intervention Closure

P8 measures decoder service time on real detector records. P10 turns that trace
into a paired runtime intervention by applying deadline budgets:

```text
deadline_us in {4, 5, 6, 7, 8}
```

If a measured decoder service time exceeds the budget, P10 treats the correction
as late and drops the prediction to a default no-flip correction. It then
computes paired rescued and induced failures over the same shots.

This is a measured-service-time runtime intervention closure. It is still not a
live quantum-control-stack trace.

## Paired Significance

P10 reports exact paired tests over discordant transitions:

```text
rescued_failure_count
        vs.
induced_failure_count
```

The output is:

```text
data/results/p10_effect_significance.csv
```

For each real decoder-pathway condition, decoder-prior variant, and measured
runtime deadline, P10 computes:

```text
McNemar exact p-value
Holm-adjusted p-value
effect direction
significance after Holm correction at 0.05
```

Holm correction is applied within each analysis scope, such as:

```text
real_decoder_pathway
real_decoder_prior
measured_runtime_deadline
external_qec3v5_decoder_pathway
google_rl_qec_v2_decoder_pathway
```

This keeps the main real-record decoder-pathway, prior, runtime-replay, and
external-replication claims from being jointly over-penalized by unrelated
families of tests.

This makes the core FailureOps claim sharper: attribution is not only an
aggregate LFR delta; it is a paired rescued/induced transition imbalance over
the same shot records.


```text

```

The audit explicitly separates what the artifact can support without a real

## EuroSys-Facing Digest

P10 also writes a compact reviewer-facing digest:

```text

```

This digest is meant to read like a EuroSys result spine rather than a raw
artifact dump. It compresses the current package into seven story units:

```text
main real-record result

expanded same-family stability

```


## External Replication

P10 includes two external-data paths that avoid the 112 GB threshold dataset:

```text
Google RL QEC v2 expanded public real-record corpus
Google qec3v5 2022 surface-code scaling dataset
ETH Zurich 2020 public surface-code figure dataset
DAQEC/IBM aggregate hardware validation dataset
```

Run the expanded Google RL QEC v2 real-record sweep:

```bash
conda run -n failureops --no-capture-output python scripts/35_run_google_rl_qec_v2_evidence.py
```

This writes:

```text
data/results/p10_google_rl_qec_v2_decoder_effect_matrix.csv
data/results/p10_google_rl_qec_v2_decoder_effect_aggregate.csv
data/results/p10_google_rl_qec_v2_rescue_induction_features.csv
data/results/p10_google_rl_qec_v2_paired_vs_unpaired_variance.csv
```

The default v2 run uses a paired decoder/prior intervention:

```text
tesseract_decoder_with_si1000_prior
        vs.
tesseract_decoder_with_frequency_calibrated_prior
```

P10 treats this as expanded same-family public real-record evidence rather than
independent external replication.

In the paper-facing claim audit, this v2 path is used as a same-family extension
check: broad net-rescuing sign stability over the expanded corpus, with
negative mean paired delta retained in each observed control-mode and basis
subgroup. It is not used to claim that every v2 condition is individually
significant or net-rescuing.

Run the qec3v5 shot-level replication:

```bash
conda run -n failureops --no-capture-output python scripts/32_run_qec3v5_external_replication.py
```

This writes:

```text

data/results/p10_qec3v5_external_decoder_effect_aggregate.csv
```


Run the public Google decoder-priors sweep:

```bash
conda run -n failureops --no-capture-output python scripts/36_run_google_decoder_priors_evidence.py
```

This writes:


The default run keeps the decoder backend fixed to `correlated_matching`,
treats `dem_simple` as the baseline prior, and compares against matched learned
prior variants plus `dem_correlations`. The resulting CSV is now the default
`--prior-effects` input of the P10 main evidence script.

At the current full-corpus run, this prior path contributes:

```text
5724 paired prior-effect rows
4879 / 5724 bootstrap CIs excluding zero
negative-delta fraction in [0.960168, 0.997379] across the three default prior families
```

Run independent non-Google sanity checks:

```bash
conda run -n failureops --no-capture-output python scripts/33_run_external_sanity_checks.py
```

This writes:

```text
data/results/p10_external_sanity_checks.csv
```

The ETH dataset is classified as public independent QEC evidence but not
FailureOps-attribution-compatible because it contains figure-level CSVs instead
of shot-level detector records. DAQEC/IBM is classified as aggregate/session
sanity evidence because it reports paired session and condition summaries, not
shot-level detector records.

## Scope

P10 strengthens the EuroSys evidence story but does not solve every main-track
requirement. It still does not provide:

```text
live hardware feedback timing
real control-stack queue traces
independent public QEC datasets beyond the current Google RL QEC records
production decoder scheduling
```

The no-machine framing should therefore be:

```text
supported:
  paired FailureOps methodology
  public real QEC detector-record evaluation
  external qec3v5 real-record replication
  real decoder-pathway and prior interventions
  measured decoder-runtime replay
  controlled runtime deadline intervention closure
  independent non-Google aggregate/figure-level sanity boundaries

not claimed:
  live hardware feedback timing
  production control-stack queueing attribution
  hardware idle exposure caused by measured decoder latency
```
