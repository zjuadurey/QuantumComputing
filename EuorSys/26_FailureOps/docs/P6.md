# P6: Cross-Mode Evaluation

## Purpose

P6 stops adding new FailureOps mechanisms and instead evaluates the mechanisms
from P5 through one unified experiment interface.

P5 introduced three input surfaces:

```text
P5a: runtime trace mode
P5b: layered event records
P5c: surface-code pilot
```

P6 asks whether intervention attribution remains meaningful across those modes.

## Unified Mode Schema

Each mode is configured as:

```text
mode_id,backend,runtime_source,event_model,workloads,stress_levels
```

Supported values:

```text
backend:
  repetition
  surface

runtime_source:
  proxy
  trace

event_model:
  detector
  layered
```

Example:

```text
repetition_trace_layered,repetition,trace,layered,memory_x|idle_heavy_memory,high
```

## Run P6

```bash
conda run -n failureops --no-capture-output python scripts/25_run_p6_cross_mode.py \
  --config configs/p6_cross_mode.yaml
```

## Outputs

Per-mode artifacts are written under:

```text
data/results/p6/
```

Each mode produces:

```text
<mode>_baseline_runs.csv
<mode>_intervened_runs.csv
<mode>_pairing_validation.csv
<mode>_paired_effects.csv
<mode>_manifest.json
```

P6 summary outputs:

```text
data/results/p6_mode_summary.csv
data/results/p6_intervention_stability.csv
data/results/p6_rank_stability.csv
```

## What P6 Measures

P6 reports:

```text
per-mode row counts
per-mode invalid pairing count
strongest intervention per mode
intervention effect spread across modes
mean paired delta LFR per intervention
top-mode intervention effects
top-3 rank overlap across modes
pairwise rank distance across modes
```

## Scope

P6 is not a new backend stage. It is a cross-mode evaluation harness.

It still does not implement:

```text
large benchmark automation
formal hypothesis testing
real runtime trace collection
production decoder scheduling
full surface-code sweep
```
