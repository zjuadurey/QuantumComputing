# P2.5: P2 Robustness And Interpretability

## Purpose

P2.5 strengthens the P2 runtime/system intervention result.

It does not add a new simulator, scheduler, QEC code, or architecture. It keeps
the existing P2 Stim/PyMatching backend and runtime proxy, then adds:

```text
multi-seed repeat analysis
runtime stress sweep
failure-pattern shift analysis
intervention sanity checks
summary plots
```

P2.5 is meant to answer:

```text
Are P2 intervention rankings stable across seeds?
Do runtime interventions become stronger under higher runtime stress?
Do interventions shift failure patterns in the expected direction?
Are the P2 results more than a one-seed artifact?
```

## Multi-Seed Repeat Summary

`p2_5_repeated_summary.csv` aggregates P2 attribution across seeds.

Fields:

```text
intervention
num_seeds
mean_baseline_lfr
mean_intervened_lfr
mean_absolute_delta_lfr
std_absolute_delta_lfr
ci95_absolute_delta_lfr
mean_rescue_rate
std_rescue_rate
```

The CI is:

```text
1.96 * std / sqrt(num_seeds)
```

Current default result:

```text
num_seeds = 10
num_shots_per_seed = 1000
mean_baseline_lfr = 0.0827
```

Strongest current mean deltas:

```text
remove_measurement_noise:       -0.0551 ± 0.0051
eliminate_decoder_backlog:      -0.0550 ± 0.0051
remove_data_noise:              -0.0533 ± 0.0052
increase_decoder_capacity_2x:   -0.0482 ± 0.0039
reduce_idle_exposure_50pct:     -0.0418 ± 0.0042
remove_decoder_timeout:         -0.0401 ± 0.0061
```

This supports the P2 claim that runtime/system interventions are competitive
with physical-noise interventions under the stressed P2 setting.

## Runtime Stress Sweep

`p2_5_stress_sweep_summary.csv` runs low, medium, and high runtime stress.

Stress settings:

```text
low:
  decoder_timeout_base_rate = 0.005
  decoder_capacity = 6.0
  synchronization_slack = 0.60

medium:
  decoder_timeout_base_rate = 0.020
  decoder_capacity = 3.0
  synchronization_slack = 0.25

high:
  decoder_timeout_base_rate = 0.040
  decoder_capacity = 2.0
  synchronization_slack = 0.10
```

Each stress level currently uses:

```text
num_seeds = 3
num_shots_per_seed = 1000
```

Expected qualitative behavior is visible:

```text
low stress:
  runtime interventions are small

medium stress:
  runtime interventions become visible

high stress:
  eliminate_decoder_backlog becomes the strongest intervention
```

Current high-stress examples:

```text
eliminate_decoder_backlog:      -0.0910
reduce_idle_exposure_50pct:     -0.0637
increase_decoder_capacity_2x:   -0.0610
remove_decoder_timeout:         -0.0560
```

## Failure-Pattern Shift

`p2_5_pattern_shift_summary.csv` analyzes how intervention outputs change
failure patterns.

For each intervention it reports:

```text
dominant_baseline_failure_pattern
dominant_intervened_failure_pattern
rescued_failures_by_baseline_pattern
baseline_pattern_distribution
intervened_pattern_distribution
timeout_correlated_delta_count
idle_correlated_delta_count
syndrome_burst_delta_count
syndrome_decoding_mismatch_delta_count
```

Current examples:

```text
remove_decoder_timeout:
  timeout_correlated_delta_count = -72

eliminate_decoder_backlog:
  timeout_correlated_delta_count = -53
  idle_correlated_delta_count = -20

reduce_idle_exposure_50pct:
  idle_correlated_delta_count = -21
```

This gives a simple behavioral explanation for the LFR deltas.

## Sanity Checks

`p2_5_sanity_checks.csv` records small interpretability guardrails.

Current checks:

```text
remove_data_noise reduces total failures
remove_measurement_noise reduces total failures
remove_decoder_timeout reduces timeout_correlated failures
eliminate_decoder_backlog reduces runtime pressure patterns
increase_decoder_capacity_2x reduces runtime pressure patterns
reduce_idle_exposure_50pct reduces idle-correlated failures or total failures
```

All six current checks pass.

The data and measurement checks are intentionally phrased as total-failure
checks because the current P2 pattern vocabulary does not distinguish
`data_dominated` and `measurement_dominated` failures.

## Limitations

P2.5 still does not implement:

```text
real decoder queues
real scheduler simulation
surface-code case studies
exact physical-event replay for noise-side interventions
data-dominated or measurement-dominated QEC pattern labels
formal statistical hypothesis testing
```

The multi-seed CI is a robustness signal, not a theorem.

The stress sweep uses a transparent runtime proxy over the existing QEC backend.
It should be interpreted as systems-oriented evidence for the FailureOps
workflow, not as a claim about a specific real machine.
