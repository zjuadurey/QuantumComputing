# P3: Systems Evaluation Harness

## Purpose

P3 turns the P2/P2.5 runtime-proxy result into a more systems-oriented
evaluation harness for a EuroSys-style paper.

It still does not build a full FTQC runtime, scheduler, or decoder. Instead, it
adds a lightweight decoder-service model and evaluates paired counterfactual
interventions across:

```text
workload shape
runtime stress level
seed
shot
intervention class
```

The goal is to test whether logical failure behavior is sensitive to
system-controllable factors such as decoder workers, service rate, queueing,
deadline policy, timeout, and idle exposure.

## Runtime Service Model

P3 keeps the Stim/PyMatching repetition-code backend, then adds a small
decoder-service trace to each shot:

```text
detector events
        ↓
scaled detector load
        ↓
bounded decoder workers
        ↓
queue wait and service time
        ↓
deadline miss / queue overflow / timeout
        ↓
idle exposure and runtime idle flip
        ↓
logical failure recomputation
```

Important runtime fields:

```text
decoder_workers
decoder_service_rate
decoder_queue_depth
decoder_deadline
decoder_arrival_time
decoder_start_time
decoder_finish_time
decoder_latency
decoder_deadline_miss
decoder_queue_overflow
decoder_timeout
decoder_backlog
runtime_stall_rounds
idle_exposure
runtime_idle_flip
```

## Workloads

P3 currently includes five small workload shapes:

```text
memory_x
memory_z
burst_sensitive_memory
high_detector_load_memory
idle_heavy_memory
```

These are not new quantum compilers or new codes. They are named configurations
of the existing backend and runtime service knobs.

## Interventions

Noise-side interventions:

```text
remove_data_noise
weaken_data_noise_50pct
remove_measurement_noise
weaken_measurement_noise_50pct
remove_idle_noise
weaken_idle_noise_50pct
```

Runtime/system interventions:

```text
remove_decoder_timeout
increase_decoder_workers_2x
increase_decoder_service_rate_2x
remove_decoder_queueing
relax_decoder_deadline_2x
reduce_idle_exposure_50pct
prioritize_high_weight_syndromes
```

Runtime interventions are paired counterfactuals over the same detector record,
observable, decoder prediction, workload, seed, and shot.

Noise interventions are paired event-record transformations. They weaken the
same detector event record rather than launching an unrelated Monte Carlo run.
This is still a proxy, not exact physical-event replay.

## Run P3

Small validation run:

```bash
conda run -n failureops --no-capture-output python scripts/16_run_p3_eval_matrix.py \
  --workloads memory_x,idle_heavy_memory \
  --stress-levels low,high \
  --num-seeds 2 \
  --num-shots-per-seed 100
```

Plot the generated summaries:

```bash
conda run -n failureops --no-capture-output python scripts/17_plot_p3_results.py
```

Larger paper-oriented run:

```bash
conda run -n failureops --no-capture-output python scripts/16_run_p3_eval_matrix.py \
  --workloads memory_x,memory_z,burst_sensitive_memory,high_detector_load_memory,idle_heavy_memory \
  --stress-levels low,medium,high \
  --num-seeds 5 \
  --num-shots-per-seed 500
```

## What P3 Adds Over P2.5

P3 adds:

```text
multi-workload evaluation
explicit decoder-service trace fields
runtime queueing/deadline/service-rate interventions
baseline method comparison
rank-stability analysis
one-shot eval-matrix runner
paper-facing P3 plots
```

P3 is meant to support a stronger systems claim:

> FailureOps can identify when logical failure behavior is sensitive to
> controllable decoder-service and runtime policies, not only physical noise.

## Limitations

P3 still does not implement:

```text
real decoder queue traces from a production runtime
real multi-worker decoder implementation
surface-code case studies
exact physical-event replay for all noise interventions
formal hypothesis testing
large artifact-optimized experiment automation
```

Those are the main remaining gaps before a EuroSys main-track submission.
