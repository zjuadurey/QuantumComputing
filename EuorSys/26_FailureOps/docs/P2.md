# P2: Runtime/System Interventions

## Purpose

P2 makes runtime and system factors first-class FailureOps interventions.

P0 established the intervention pipeline with a toy model. P1 connected the
pipeline to a lightweight Stim/PyMatching QEC backend. P2 keeps the same backend
and attribution table, but expands the runtime model so FailureOps can ask:

> Is logical failure behavior more sensitive to physical noise, decoder timeout,
> decoder backlog, decoder capacity, idle exposure, or synchronization slack?

P2 is still intentionally small. It is not a real runtime scheduler, decoder
resource manager, or FTQC architecture simulator.

From the repository root:

```bash

  --num-shots 1000 \
  --distance 5 \
  --num-rounds 8 \
```

## Runtime Fields

P2 extends the QEC-backed baseline record with system-side fields:

```text
decoder_capacity
decoder_backlog
synchronization_slack
decoder_timeout
decoder_delay
idle_exposure
runtime_idle_flip
```

These fields are deterministic post-processing signals derived from:

```text
detector_event_count
distance
num_rounds
idle_error_rate
decoder_timeout_base_rate
decoder_capacity
synchronization_slack
shot seed
```

The runtime model is deliberately simple:

- More detector events increase decoder load.
- Load above `decoder_capacity` creates `decoder_backlog`.
- Backlog increases decoder delay and timeout probability.
- Lower synchronization slack increases idle exposure.
- Idle exposure may flip the effective logical observable through
  `runtime_idle_flip`.

This is a controllable systems proxy, not a real decoder scheduler.

## Interventions

P2 compares noise-side interventions against runtime/system interventions in one
attribution table.

Noise-side interventions:

```text
remove_data_noise
weaken_data_noise_50pct
remove_measurement_noise
weaken_measurement_noise_50pct
remove_idle_noise
weaken_idle_noise_50pct
```

Runtime/system interventions:

```text
remove_decoder_timeout
relax_timeout_policy
reduce_decoder_delay_50pct
reduce_idle_exposure_50pct
increase_decoder_capacity_2x
eliminate_decoder_backlog
increase_synchronization_slack
```

## Pairing Semantics

Runtime/system interventions are strictly paired:

```text
same shot_id
same seed
same detector_events
same observable_flip
same decoder_prediction
same QEC backend outcome
```

Only the runtime/system policy is changed, then the final logical failure
behavior is recomputed.

Noise-side interventions follow the P1 semantics:

```text
same shot_id
same batch seed
same backend configuration except the intervened noise parameter
```

They are seed-stable backend comparisons, not exact physical-event replay.

## Interpretation

P2 supports statements such as:

```text
Under this stressed repetition-code memory workload, logical failure behavior is
highly sensitive to decoder backlog and decoder capacity. These runtime factors
are comparable to, and in this run slightly stronger than, some physical-noise
interventions.
```

That statement is not a physical claim about a real machine. It is a systems
claim about the FailureOps attribution workflow.

## Current Limitations

P2 does not yet implement:

```text
real decoder service queues
real wall-clock decoder latency
hardware calibration data
multi-workload runtime contention
operation pacing policies
surface-code runtime case studies
exact physical-event replay for noise-side interventions
```

The runtime model is still a transparent toy systems layer over a real
Stim/PyMatching detector-event backend.

## Next Steps
