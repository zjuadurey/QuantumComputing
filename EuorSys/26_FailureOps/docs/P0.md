# P0: Toy Intervention Pipeline

## Purpose

P0 is the first runnable FailureOps prototype.

Its goal is to make the core workflow executable:

```text
generate baseline logical execution records
        ↓
apply paired counterfactual interventions
        ↓
compare baseline and intervened outcomes
        ↓
compute attribution metrics
        ↓
write CSV summaries and a simple figure
```

P0 is intentionally a toy model. It is not a physical QEC simulator, a decoder,
a threshold experiment, or a realistic FTQC runtime.

The purpose of P0 is to test the FailureOps abstraction:

> Which intervenable factors is the logical failure behavior sensitive to?

## Environment

Use the conda environment recorded in `docs/env.md`:

```bash
conda run -n failureops --no-capture-output python ...
```

The plot script prefers `matplotlib` when available. If `matplotlib` is not
installed, it falls back to ImageMagick `convert`.

## Run The Full P0 Pipeline

From the repository root:

```bash
conda run -n failureops --no-capture-output python scripts/01_generate_runs.py \
  --num-shots 1000 \
  --num-rounds 10 \
  --num-operations 5 \
  --data-error-rate 0.03 \
  --measurement-error-rate 0.02 \
  --idle-error-rate 0.01 \
  --decoder-timeout-rate 0.02 \
  --seed 42 \
  --output data/results/baseline_runs.csv

conda run -n failureops --no-capture-output python scripts/02_apply_interventions.py \
  --input data/results/baseline_runs.csv \
  --output data/results/intervened_runs.csv

conda run -n failureops --no-capture-output python scripts/03_analyze_attribution.py \
  --input data/results/intervened_runs.csv \
  --output data/results/attribution_summary.csv

conda run -n failureops --no-capture-output python scripts/04_plot_results.py \
  --input data/results/attribution_summary.csv \
  --output figures/intervention_delta_lfr.png \
  --title "FailureOps P0 intervention sensitivity"
```

Expected row counts for the command above:

```text
baseline_runs.csv:      1000 data rows
intervened_runs.csv:    9000 data rows
attribution_summary.csv:   9 data rows
```

The intervention file has one row per baseline shot per intervention.

## Baseline Records

`baseline_runs.csv` stores one logical execution shot per row.

Important fields include:

```text
shot_id
seed
num_rounds
num_operations
data_error_count
measurement_error_count
idle_error_count
decoder_timeout
decoder_delay
idle_exposure
error_events
logical_failure
failure_round
failure_region
failure_operation
failure_mode
failure_pattern
```

`error_events` is a compact JSON list of toy events. Each event has:

```text
event_type
round_id
operation_id
region_id
weight
```

## Toy Failure Rule

P0 uses a deterministic toy failure score:

```text
logical_failure = failure_score >= threshold
```

The score is a weighted sum of event categories:

```text
data_error
measurement_error
idle_error
decoder_timeout
decoder_delay
idle_exposure
```

This rule is intentionally transparent. It is only a proxy for making paired
intervention analysis executable.

After every intervention, P0 recomputes:

```text
logical_failure
failure_round
failure_region
failure_operation
failure_mode
failure_pattern
```

## Interventions

P0 implements both noise-side and runtime-side interventions.

Noise-side interventions:

```text
remove_data_errors
remove_measurement_errors
remove_idle_errors
weaken_data_errors_50pct
weaken_measurement_errors_50pct
weaken_idle_errors_50pct
```

Runtime-side interventions:

```text
remove_decoder_timeout
reduce_decoder_delay_50pct
reduce_idle_exposure_50pct
```

Each intervention is paired with the original baseline shot. The script does
not resample a new Monte Carlo execution for each intervention.

The intended question is:

> Would this same execution still fail if one factor were removed, weakened, or changed?

## Attribution Summary

`attribution_summary.csv` reports one row per intervention.

Important fields:

```text
intervention
num_shots
baseline_logical_failure_rate
intervened_logical_failure_rate
absolute_delta_lfr
relative_delta_lfr
baseline_failure_count
intervened_failure_count
rescued_failure_count
new_failure_count
rescue_rate
new_failure_rate
dominant_baseline_failure_pattern
dominant_intervened_failure_pattern
```

Interpretation:

```text
absolute_delta_lfr =
  intervened_logical_failure_rate - baseline_logical_failure_rate
```

A more negative value means the intervention reduced logical failure rate more.

`rescued_failure_count` counts shots that failed at baseline but succeeded after
the intervention.

`new_failure_count` counts shots that succeeded at baseline but failed after the
intervention. For removal or weakening interventions this should usually be
zero in the toy model, but P0 reports it explicitly instead of hiding it.

## Current Limitations

P0 does not implement:

```text
real surface-code simulation
real decoder integration
Stim or PyMatching backend
threshold estimation
large benchmark suite
full runtime scheduler
causal graph inference
```

The toy event model and toy failure rule are not meant to support physical QEC
claims.

P0 only claims that the FailureOps workflow can be made executable:

```text
baseline execution
        vs.
paired intervention
        ↓
sensitivity attribution
```

## Next Steps

The natural next step is P1:

```text
connect FailureOps to a lightweight QEC backend
```

Candidate directions:

```text
Stim repetition-code memory
small surface-code memory experiment
PyMatching decoding
seed-stable syndrome/logical outcome records
paired or replayable intervention analysis
```

P1 should preserve the same output style as P0:

```text
baseline records
intervened records
attribution summary
simple figures
```
