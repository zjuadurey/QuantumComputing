## 本轮目标

本轮主要完成两件事：

1. 继续推进 `dev32` 开发线，把 `NSO` 与 `FNO` 的主力配置补成更可信的对照。

## 代码修改概览

### 1. `NSO` leakage regularization 从“记录项”改成“真正影响学习的项”

问题定位：

- 旧版训练里使用的 `shadow_pool_leakage` 对同一个 sample 的候选池几乎是常数。
- 这会导致 `loss = loss_task + lambda * leakage` 中的 leakage 项对候选选择几乎没有梯度影响。
- 直接现象是 `nso` 与 `nso_highleak` 的结果几乎完全一致，说明调大 `w_leakage` 并没有改变学习行为。

修正方式：

- 在 `code/shiftflow/bench/math_utils.py` 中，为每个候选池新增
  - `outside_pool_bra_leakage`
  - `pair_bra_coupling_sq`
- 在 `code/shiftflow/bench/data.py` 中把上述张量写入数据集并在 dataloader 中返回。
- 在 `code/shiftflow/bench/models.py` 中引入依赖软选择权重的 differentiable set-level leakage surrogate：
  - 惩罚选中参考模式向候选池外泄漏
  - 惩罚选中参考模式耦合到“池内但未覆盖”的模式
- 同时加入 `selection_temperature`、`selection_entropy`、`selection_coverage_deficit` 相关输出。

效果：

- leakage 不再是摆设，而是会改变 learned dictionary 的形状。
- `nso_diverse` 相比基础版表现出更低的选择熵和更明确的子集选择行为。

### 2. `dev32` 数据生成切换到 `expm_multiply` exact backend

为避免 `N=32` 下每个 Hamiltonian 都做 dense eigendecomposition，做了如下调整：

- 在 `code/shiftflow/bench/math_utils.py` 中为 exact rollout 增加 `expm_multiply` 路径。
- 在 `code/shiftflow/bench/data.py` 中加入 Hamiltonian / potential 相关 cache，并支持 `dataset.exact_solver` 配置。

这使得 `dev32_dataset_v3` 的生成与后续评测更适合继续做开发调参与 OOD 分析。

## 2026-05-06 深夜：继续增强 `NSO`

在 `exp32` 最终结果出来后，又继续沿两条最可能的改进方向推进：

### A. Candidate-aware selection

问题判断：

- 旧版 `NSO` 的 `generator` 只看 `hamiltonian_features`，再直接输出 `num_slots × P` 个 logits。
- 这意味着它更容易学习“第几个 pool 位置更常被选中”，而不是根据 candidate 自身的 shadow 特征、leakage 和 mode 位置来选择字典。

为此在 `code/shiftflow/bench/models.py` 里增加了：

- `selection_mode = candidate_attention`
- 基于 candidate shadow trajectory 的 `candidate_time_encoder`
- 基于 `(mean_feat, last_feat, leakage, outside_leakage, mode_coords, radius)` 的 candidate key
- 基于 Hamiltonian/context 的 slot query

对应配置：

- `code/configs/nso_exp32_attn_context.yaml`

当前结果：

- 这条线的最佳验证点大约为：
  - `val density_rel_l2 ≈ 0.23294`
  - `val spectral_rel_l2 ≈ 0.23043`
  - `val energy_rel_l2 ≈ 0.00255`

解释：

- 选择行为更健康了，entropy 不再塌成极小值。
- 但就当前验证集表现看，它还没有明显超过旧版 `nso_relaxed`。

### B. Raw shadow decoder

问题判断：

- 旧版 `NSO` 在 decoder 前会把每个 slot 的 shadow feature 压缩到较小的 `slot_embed_dim`。
- 这可能形成信息瓶颈，使 readout 无法充分利用原始 shadow trajectory 中的结构信息。

为此在 `code/shiftflow/bench/models.py` 里增加了：

- `decoder_raw_features = true`
- 允许 decoder 直接吃更高维的 pooled raw shadow features
- 可选 `context_dim`，让 decoder 额外获得 `rho0 + potential_field` 的 field context

对应配置：

- `code/configs/nso_exp32_rawdecode.yaml`

当前中期信号：

- 最佳验证点暂时大约为：
  - `val density_rel_l2 ≈ 0.23275`
  - `val spectral_rel_l2 ≈ 0.23011`

解释：

- 这条线是当前最值得继续看的结构增强方向之一。
- 它说明 decoder 信息瓶颈确实可能是问题，但目前还没有形成明显领先。

### C. Robust model selection

问题判断：

- 旧版 `NSO/FNO` 都有明显的 `split_id(val) -> split_id(test)` 落差。
- 只靠 ID 验证集选模，可能会偏向对 val set 过拟合而非真正泛化好的 checkpoint。

为此在 `code/shiftflow/bench/runner.py` 中加入了：

- `train.extra_selection_splits`
- 训练时可同时评估额外的验证 split，并把它们加权纳入选模分数

当前使用的配置：

- `code/configs/nso_exp32_relaxed_robustsel.yaml`

这条线会同时参考：

- `split_id = val`
- `split_ood_alpha = val`
- `split_ood_structure = val`

当前中期信号：

- 最佳验证点暂时大约为：
  - `val density_rel_l2 ≈ 0.23267`
  - `val spectral_rel_l2 ≈ 0.22389`
  - `val energy_rel_l2 ≈ 0.00556`

解释：

- 这条线的核心价值不一定是把 val density 再压很多，而是希望选出 test / OOD 上更稳的 checkpoint。
- 从当前数值看，它是三条新增路线里最有希望改善“泛化选模偏差”的一条。

### 结论

1. `candidate-aware attention` 没有带来测试集提升，说明当前瓶颈不只是“选择器没看见 candidate 元信息”。
2. `raw shadow decoder` 也没有提升，说明简单扩大 decoder 信息通路并不能直接解决泛化问题。
3. `robust model selection` 至少没有变差，并且几乎复现了 `nso_relaxed` 的最优测试表现，但也没有形成显著超越。

因此目前最可靠的判断是：

- `NSO` 的主要问题不是单点模块配置，而更像是整体学习目标与数据分布之间的错位。
- 也就是说，现在继续小修小补某一个子模块，边际收益已经开始变小。
